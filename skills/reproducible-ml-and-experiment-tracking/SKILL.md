# Reproducible ML and Experiment Tracking

## Overview

Reproducibility and provenance for analytics and machine learning. Every other safeguard — privacy, quality, honest metrics, interpretability, monitoring — assumes you can say exactly what produced a given result: which data, which code, which environment, which random seed. Without that, a model card describes a model you can't rebuild, a monitoring alert compares against a baseline you can't recreate, and an audit has nothing to audit. This skill is the substrate: it makes results regenerable on another machine, by another person, months later, and traceable back to their inputs. In healthcare that traceability is not craftsmanship for its own sake — it is what makes a result count as evidence.

Core principle: **a result you cannot regenerate is an anecdote, not a finding.** If "it works on my machine" is the only proof, there is no proof.

## When to Use

- Starting any analysis or modeling project (set this up first, not last)
- Before sharing, publishing, or presenting any result or model
- A number, figure, or model can't be reproduced — by a collaborator or by you later
- Handing a project to someone else, or expecting to return to it after months
- Comparing experiments and needing to know which change caused which outcome
- Preparing work for audit, review, or regulatory scrutiny

**NOT for:** the domain-specific concerns of the other skills (each has its own gate); this is the connective tissue that makes their outputs trustworthy and re-runnable.

## Process: The Reproducibility Gate

### Step 1: Structure the project so inputs, code, and outputs are separable

Adopt a predictable layout so anyone can find what produced what:

```
project/
  data/raw/          # immutable inputs (gitignored; see privacy & quality skills)
  data/clean/        # derived, regenerated by code from raw
  src/ or notebooks/ # code, in run order
  models/            # serialized models, versioned
  outputs/           # figures, tables, reports (regenerated, not hand-edited)
  environment.yml /  # pinned dependencies
  requirements.txt
  README.md          # how to reproduce, end to end
```

Derived artifacts (clean data, models, figures) are *products* of code, never hand-edited — if a figure can only be recreated by manual clicking, it isn't reproducible.

### Step 2: Pin the environment

"It works on my machine" is a dependency-version statement. Capture the exact environment so it works on any machine:

```bash
# Record exact versions, not floating ranges
pip freeze > requirements.txt          # or a lockfile (pip-tools, poetry, uv)
# or conda:
conda env export --no-builds > environment.yml
```

Pin Python and key library versions explicitly (a model's behavior can change across library versions). For stronger guarantees, containerize (a Dockerfile) so the OS-level environment is captured too. Record the versions in the README so a reader knows the target before they start.

### Step 3: Control randomness with recorded seeds

Stochastic steps — train/test splits, weight initialization, shuffling, resampling, bootstrapping — must be seeded and the seed recorded, or results drift run to run:

```python
import os, random, numpy as np
SEED = 42
os.environ["PYTHONHASHSEED"] = str(SEED)
random.seed(SEED); np.random.seed(SEED)
# framework-specific: sklearn random_state=SEED; torch.manual_seed(SEED); etc.
```

Note honestly that full determinism isn't always achievable (some GPU ops, parallelism), so document what was seeded and any residual nondeterminism rather than claiming bit-for-bit reproducibility you can't guarantee.

### Step 4: Version data and models, not just code

Git versions code; it does not (and should not) hold large or sensitive data. Track the others deliberately:

- **Data:** version by reference — a content hash/checksum, a dataset version tag, or a data-versioning tool (DVC, lakeFS) — so you can prove which data produced a result without committing raw data to git (ties to the privacy skill). At minimum, record a checksum and a data dictionary.
- **Models:** serialize with a version, and store *alongside each model* the data version, code commit, hyperparameters, environment, and metrics that produced it — a model with no lineage can't be trusted or rebuilt.

### Step 5: Track experiments so changes map to outcomes

When you run many variants, memory and notebook scrollback are not a record. Log each run's inputs and outputs systematically:

- Parameters (hyperparameters, data version, seed), metrics (the ones chosen in the metrics skill), and artifacts (model, plots), each tied to the code commit.
- Use an experiment tracker (MLflow, Weights & Biases) or, for small projects, a disciplined structured log (a CSV/JSON per run) — the tool matters less than the discipline of recording enough to answer "which change caused this improvement?"

```python
# Minimal honest tracking without heavy tooling
import json, subprocess, datetime
run = {
  "timestamp": datetime.datetime.now().isoformat(),
  "git_commit": subprocess.check_output(["git","rev-parse","HEAD"]).decode().strip(),
  "seed": SEED, "data_version": "sha256:...", "params": params, "metrics": metrics,
}
json.dump(run, open(f"outputs/runs/{run['timestamp']}.json","w"), indent=2)
```

### Step 6: Separate config from code, and make the pipeline re-runnable end to end

- Keep parameters (paths, thresholds, hyperparameters) in config, not hard-coded in scattered cells, so a run is defined by code + config + data version.
- The whole pipeline should run start to finish with a single documented command (a script, `make`, or an ordered notebook run), reproducing `data/clean`, the model, and the outputs from raw. A pipeline that only works when cells are run in a remembered order is not reproducible.
- Clear notebook outputs before committing and ensure a top-to-bottom re-run works (no hidden state from out-of-order execution) — this also prevents leaking data in committed cell outputs (ties to privacy/quality skills).

### Step 7: Document how to reproduce, and record provenance

Write the README (and a `REPRODUCE.md` if needed) so a stranger can regenerate everything: prerequisites and pinned environment, how to obtain the data (or the synthetic sample) and verify its checksum, the exact command(s) to run, expected outputs, and the seed. For a model, ensure its lineage record (Step 4) ties result → data version → code commit → environment → seed. This provenance chain is what turns "here are my results" into "here is exactly how to verify my results," and it's the foundation an audit, a reviewer, or future-you stands on.

## Common Rationalizations

| Rationalization | Reality |
|---|---|
| "It works on my machine" | That's a statement about your unpinned environment, not a reproducible result. Pin versions so it works anywhere. |
| "I'll remember which version gave that number" | You won't, and a collaborator never knew. Track parameters, data version, seed, and commit per run. |
| "I set a seed, so it's fully reproducible" | Seeds cover much randomness but not all (some GPU/parallel ops). Document what's seeded and any residual nondeterminism. |
| "The data's in git, so it's versioned" | Raw/large/sensitive data shouldn't be in git. Version by checksum or a data-versioning tool, and keep raw out (privacy skill). |
| "I tuned it in the notebook, the best result is in there somewhere" | Scrollback isn't a record. Untracked experiments can't tell you which change caused which outcome. |
| "I edited the final figure to look right" | A hand-edited output can't be regenerated and can't be trusted. Outputs are products of code. |
| "Reproducibility is academic polish" | In healthcare it's the basis of evidence and audit. A result no one can regenerate or trace shouldn't influence a decision. |

## Red Flags

- No pinned environment (floating versions, or none recorded)
- No seed, or a seed that isn't recorded; results change run to run unexplained
- A model file with no record of the data, code commit, params, and environment that made it
- Results produced by running notebook cells in a remembered, undocumented order
- Hand-edited figures, tables, or "clean" data that code cannot regenerate
- Many model variants tried with no experiment log mapping changes to outcomes
- Raw or sensitive data committed to git as the "versioning" mechanism
- A README that doesn't let a stranger reproduce the headline result

## Verification

- [ ] Project structure separates immutable raw inputs, code, and regenerated outputs
- [ ] Environment pinned (lockfile/`environment.yml`, optionally a container); versions noted in README
- [ ] Randomness seeded and the seed recorded; residual nondeterminism documented honestly
- [ ] Data versioned by checksum/tag/tool without committing raw data; models stored with full lineage (data version, commit, params, env, metrics)
- [ ] Experiments tracked (tool or structured log) so each result maps to its parameters, data, seed, and commit
- [ ] Config separated from code; pipeline runs end to end from one documented command; notebooks re-run top to bottom with outputs cleared
- [ ] README/`REPRODUCE.md` lets a stranger regenerate and verify the headline result, including data access and checksum
- [ ] Provenance chain intact: result → data version → code commit → environment → seed

If any box lacks evidence, the result cannot be independently verified — and an unverifiable result, however impressive, is not yet evidence anyone should act on.
